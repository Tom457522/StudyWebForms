入力検証(Validation)

宣言的な Validator コントロールとサーバー側の最終防衛線

本日のゴール

  • 代表的な Validator コントロール(必須・形式・範囲)の役割を説明できる
  • Validator はブラウザ側とサーバー側の両方でチェックされることを理解する
  • サーバー側で Page.IsValid を必ず確認すべき理由を説明できる

ここまでの流れ

第2〜5章:コントロールの値を読み書きし、状態を保持する方法を見てきた。では「その値が正しいか」はどこでチェックするのか?

定義:「入力検証」の2段階

Validator コントロール:
入力値が要件を満たしているかをチェックする Web Forms のコンポーネント。RequiredFieldValidator / RangeValidator
Page.IsValid:
すべての Validator がチェックに合格したかを示す bool プロパティ。サーバー側の処理を行う前に「必ず」確認する必要がある。

つまり:「正しい入力だけを処理する」という防衛線を、ブラウザとサーバーの両方に引く

入力検証の2段階

ステージ1 - クライアント側(JavaScript)

ユーザーがデータを入力・送信するときに、即座にチェック。UX 改善(エラーを素早く表示)

ステージ2 - サーバー側(C#)

Page.IsValid で確認してから実際の処理(データベース保存など)を実行。最終防衛線— JavaScript は外部から改ざん可能

重要なセキュリティ原則

「ブラウザの検証」だけを信用してはいけない。悪意あるユーザーはブラウザの JavaScript を無効化したり、直接サーバーに不正データを送信できる。

代表的な Validator

RequiredFieldValidator — 必須入力チェック(空欄を禁止)
RegularExpressionValidator — 正規表現による形式チェック(メール形式など)
RangeValidator — 数値・日付の範囲チェック(0〜120 など)
ValidationSummary — すべてのエラーを1か所にまとめて表示

コードで確認①:マークアップ

06_Validation.aspx

<asp:TextBox ID="EmailTextBox" runat="server" />
<asp:RequiredFieldValidator ControlToValidate="EmailTextBox"
    ErrorMessage="メールアドレスは必須です。" />
<asp:RegularExpressionValidator ControlToValidate="EmailTextBox"
    ValidationExpression="^[^@\s]+@[^@\s]+\.[^@\s]+$" />

<asp:TextBox ID="AgeTextBox" runat="server" />
<asp:RangeValidator ControlToValidate="AgeTextBox"
    Type="Integer" MinimumValue="0" MaximumValue="120" />

各 Validator は ControlToValidate で対象を指定するだけ — 宣言的に書ける

コードで確認②:サーバー側の最終防衛線

06_Validation.aspx.cs

protected void SubmitButton_Click(object sender, EventArgs e)
{
    if (!Page.IsValid)
    {
        ResultLabel.Text = "入力エラーがあります。";
        return;
    }
    // ここに来るのは Page.IsValid == true のときだけ
}

なぜサーバー側でも確認するのか

ブラウザの JavaScript は無効化・改ざんが可能。クライアント側の検証だけを信用してはいけない。

デモで確認するポイント

いつどこを見る何が分かるか
未入力のまま「登録」メール欄下のエラーRequiredFieldValidator が送信をブロック
不正な形式のメールEmailFormat のエラーRegularExpressionValidator が形式チェック
範囲外の年齢(例:200)AgeRange のエラーRangeValidator が範囲チェック
すべて正しい値で「登録」結果ラベルPage.IsValid が true になり登録処理が実行される

よくある誤解・注意点

  • 「Validator を置いただけで安全」という誤解 → サーバー側で Page.IsValid のチェックを忘れると素通りする
  • JavaScript 無効環境でもサーバー側チェックが最後の砦になる
  • Display="Dynamic" の指定でエラー表示時のレイアウト崩れを防いでいる

まとめ

  • Validator コントロールは宣言的に入力チェックを追加できる
  • チェックはブラウザ側・サーバー側の両方で行われる
  • サーバー側では必ず Page.IsValid を確認してから処理を進める
