では「ボタンを押す」という操作は、裏側で何をトリガーしているのか? — それが PostBack です。

定義:「PostBack」と「IsPostBack」

PostBack:
ユーザーがボタンを押したときに発生する「フォーム送信」。同じ URL へページ全体がフォームデータとして再送信されること。新しいページに遷移しない。
IsPostBack:
現在のリクエストが「初回の表示」なのか「PostBack(再送信)」なのかを判定する bool プロパティ。if (!IsPostBack) { 初期化 } という使い方が基本パターン。

つまり:同じページが何度も呼ばれるので、「初回だけ初期化する」という分岐が必須

PostBack とは何か

同じ画面
→
フォーム送信(同じURL)
→
サーバーで処理
→
同じ画面を再生成
  • ブラウザは別ページに遷移しない — 同じ URL にフォーム全体を再送信する
  • ページ内のどのボタンを押しても、一度ページ全体がサーバーに送信される
  • この再送信であることを判定するプロパティが Page.IsPostBack

コードで確認①:IsPostBack 分岐

03_PostBack.aspx.cs

protected void Page_Load(object sender, EventArgs e)
{
    if (!IsPostBack)
    {
        ViewState["Count"] = 0;
    }
    RenderCount();
}

初回表示のときだけ 0 に初期化し、それ以外(PostBack)では前回値をそのまま使う — この if (!IsPostBack) パターンは第7章・第8章でも再登場します。

コードで確認②:イベントハンドラ

protected void CountButton_Click(object sender, EventArgs e)
{
    ViewState["Count"] = ((int)ViewState["Count"]) + 1;
    RenderCount();
}

protected void ResetButton_Click(object sender, EventArgs e)
{
    ViewState["Count"] = 0;
    RenderCount();
}

デモで確認するポイント

いつどこを見る何が分かるか
初回表示ラベルの表示IsPostBack=False / Count=0
「カウントを増やす」押下後ラベルの表示IsPostBack=True になり、Count が加算される
ボタンを押している間ブラウザのアドレスバーURL が変化しない(遷移ではなく再送信)

よくある誤解・注意点

  • 「ボタンごとに専用のURLがある」わけではない → 全ボタンが同じ URL への PostBack
  • if (!IsPostBack) を書き忘れると、ボタンを押すたびに初期化処理が実行されてしまう
  • Count の保持には ViewState を使っている(第4章で詳しく扱う)

まとめ

  • PostBack = 同じ URL への フォーム再送信
  • IsPostBack で「初回表示」か「PostBack後」かを判定する
  • if (!IsPostBack) { 初期化 } は頻出パターン
