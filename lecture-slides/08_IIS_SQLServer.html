IIS / アプリケーションプール / SQL Server

すべての仕組みが動いている「土台」を見る

本日のゴール

  • Web Forms アプリが IIS / アプリケーションプール(w3wp.exe)上で動作していることを説明できる
  • これまで見てきた ViewState / Session / DB接続がこの構成の上で動いていることを理解する
  • Web.config の接続文字列がコードからどう読み取られるかを説明できる

全章の振り返り

第1章 Page は毎回作り直される
第2章 コントロールは最終的にHTML
第3章 PostBack で同じ画面に再送信
第4章 ViewState はクライアント保存
第5章 Session/Applicationは共有範囲が違う
第6章 Validatorとサーバー側の最終確認
第7章 GridViewとRepositoryパターン

これら全部が、実は1つのプロセス(w3wp.exe)の中で動いています。

↓ connectionStrings

var setting = ConfigurationManager.ConnectionStrings["StudyDb"];
var connection = setting != null ? setting.ConnectionString : "未設定";

Environment.MachineName          // 実行マシン名
HttpRuntime.AppDomainAppPath      // IIS上の物理フォルダ
AppDomain.CurrentDomain.FriendlyName  // AppDomain名

接続文字列は Web.config の <connectionStrings> に定義 → ConfigurationManager 経由で読み取る

デモ・確認するポイント

どこを見る何が分かるか
MachineName の表示実際にどのマシン上のプロセスで実行されているか
Application PathIIS Manager の「物理パス」と一致すること
AppDomain1つの w3wp.exe 内で、サイトごとに AppDomain が分かれていること
StudyDb 接続文字列Web.config の設定がコードから読み取れること
IIS Manager のアプリケーションプール.NET CLR バージョン・実行アカウント・リサイクル設定

講義ポイント・注意点

  • IIS サイトはURLと物理フォルダを対応付ける
  • アプリケーションプールは実行プロセス・.NET CLRバージョン・実行アカウント・リサイクル設定に関係する
  • 1つのアプリケーションプールで問題が起きても、別プールのサイトには影響しない(プロセス分離)
  • 本番環境では接続文字列・実行権限・ログ・エラー表示の扱いを開発環境と分ける

まとめ:全8章を通して

Web Forms は「古い技術」ではなく、Page ライフサイクル・状態管理・検証・データバインドという明確なモデルを持つフレームワーク。その全てが IIS / アプリケーションプール / SQL Server という現実のインフラの上で動いている。

現場の既存システムを読むときは、「今どのイベントが動いているか」「状態はどこに保存されているか」「どのプロセス上で動いているか」の3点を意識すると理解が早まります。
