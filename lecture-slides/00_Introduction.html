今学ぶ意義: Web Forms の概念(ライフサイクル・状態管理・バリデーション)は、MVC や他のフレームワークの理解を深める基礎になる

アーキテクチャの比較

観点 Web Forms MVC / .NET Core
状態管理 ViewState(クライアント) / Session(サーバー) 主に Session / Cookie(明示的)
コンポーネント サーバーコントロール(asp:Button等) HTML タグ直接記述
イベント処理 Page_Load / Click など「ライフサイクル」 HTTP メソッド(GET/POST)ベース
ルーティング ファイル構造(*.aspx = URL) 明示的なルーティング設定
学習曲線 高速(デスクトップアプリ的な直感) 低速(HTTP・HTML の理解必須)

用語ガイド

Page オブジェクト: 1リクエストごとに作られる ASP.NET ページの実行主体(第1章で詳しく)
PostBack: 同じページへの再送信(フォーム送信)(第3章で詳しく)
ViewState: ページの状態をクライアント側に hidden フィールドとして保存する仕組み(第4章で詳しく)
Session: ユーザーごとの情報をサーバー側で保持する仕組み(第5章で詳しく)
サーバーコントロール: <asp:Button> など、C# コードから操作できるコンポーネント(第2章で詳しく)

Web Forms が「古い」と言われる理由

  • ステートフル設計: HTTP はもともとステートレス。ViewState で無理に状態を持たせるのは、Web の本質と相容れない
  • 隠蔽が強すぎた: PostBack・ViewState などが裏側で自動化されすぎて、HTTP の理解が後回しになりやすい
  • スケーリングの課題: Session や ViewState の管理は、大規模分散環境では複雑になる

だからこそ今学ぶ意義がある

「Web Forms の限界」を理解することで、MVC や Node.js / Python などの「ステートレス設計」の必然性が見えてくる

このシリーズで習得すること

  • HTTP / Web の基本: ステートレスな HTTP の上に、状態をどう乗せるのか
  • フレームワークの設計思想: 異なる時代の3つのフレームワーク(Forms / MVC / Core)の比較から
  • 実装パターン: ライフサイクル・状態管理・バリデーションなど、すべての Web フレームワークに共通する概念
  • レガシー理解: 実際のプロジェクトで Web Forms に遭遇したときの読み解く力
