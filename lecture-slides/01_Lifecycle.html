ページライフサイクル

Page インスタンスと実行順序の基本を理解する

本日のゴール

  • Web Forms では 1リクエストごとに Page オブジェクトが新しく作られる ことを説明できる
  • Page_Init → Page_Load → イベント処理 → Page_PreRender という実行順序を説明できる
  • 状態を持ち越すには ViewState など明示的な仕組みが必要な理由を説明できる
この章は今後すべての章(PostBack・ViewState・Validationなど)の土台になります。

まず定義:「Page オブジェクト」とは何か

Page オブジェクト:
1つのリクエストに対して、IIS / ASP.NET ランタイムが生成する C# クラスのインスタンス。このインスタンスの上で、ライフサイクルイベント(Page_Init, Page_Load など)が順番に実行される。
ライフサイクル:
1つのリクエストが終わるまでの間に、Page が通過する決まった順序のステージ。Page_Init → Page_Load → Event → Page_PreRender → Render が常に同じ順序で実行される。

つまり:毎回 Page は「生まれて」「成長して」「消える」一連の流れを通る

なぜここから始めるか

ありがちな誤解

「コードの中で持っている変数の値は、次の操作でもそのまま残っている」

実際の仕組み

Web Forms は操作のたびに Page を1から作り直す。値を残すには明示的な保存が必要

この誤解を放置すると、PostBack / ViewState / Validation すべての章でつまずく原因になります。

大原則:1リクエスト = 1つの Page インスタンス

  • 唯一「持ち越せる」のが ViewState / Session などの明示的な状態保存の仕組み(第4〜5章)

比較:Web Forms vs MVC / ASP.NET Core

Web Forms(このシリーズ)

設計思想: デスクトップアプリに近い「イベント駆動」

Page の流れ: 毎リクエストで new Page() → ライフサイクル → 破棄

状態管理: ViewState(クライアント) / Session(サーバー)が自動化

MVC / .NET Core

設計思想: HTTP の本質に従う「ステートレス」

Page の流れ: リクエスト = Controller メソッド 1つの呼び出し

状態管理: Cookie / Session を明示的に操作(自動化されない)

Web Forms を理解することで、MVC がなぜそのような設計になったのかが見えてくる

ライフサイクルイベントの順序

Page_Init
→
(ViewState 復元)
→
Page_Load
→
クリックイベント
→
Page_PreRender
→
Render
  • Page_Init:コントロールのID・階層は決まるが、ViewState はまだ復元されていない
  • Page_Load:ViewState 復元後。初期データ設定などによく使う
  • ボタンなどのイベント処理は Load の後、PreRender の前に実行される
  • Page_PreRender:HTML 生成直前の最終調整。この後の変更は反映されないことがある

コードで確認①:ログの保存先

01_Lifecycle.aspx.cs ログは ViewState に保存されるからこそ、Page が作り直されても消えない

private List<string> Log
{
    get
    {
        if (ViewState["LifecycleLog"] == null)
        {
            ViewState["LifecycleLog"] = new List<string>();
        }
        return (List<string>)ViewState["LifecycleLog"];
    }
}

コードで確認②:各イベントでの記録(前半)

protected void Page_Init(object sender, EventArgs e)
{
    Log.Add($"{DateTime.Now:HH:mm:ss.fff} Page_Init");
}

protected void Page_Load(object sender, EventArgs e)
{
    Log.Add($"{DateTime.Now:HH:mm:ss.fff} Page_Load / IsPostBack={IsPostBack}");
}

※ メソッド名がそのままライフサイクルのイベント名に対応している

コードで確認②:各イベントでの記録(後半)

protected void ReloadButton_Click(object sender, EventArgs e)
{
    Log.Add($"{DateTime.Now:HH:mm:ss.fff} ReloadButton_Click");
}

protected void Page_PreRender(object sender, EventArgs e)
{
    Log.Add($"{DateTime.Now:HH:mm:ss.fff} Page_PreRender");
    LifecycleLog.DataSource = Log;
    LifecycleLog.DataBind();
}

Page_PreRender で初めて DataBind() が呼ばれ、ここまでのログがまとめて画面に反映される

デモで確認するポイント

いつどこを見る何が分かるか
初回表示 実行ログの並び Page_Init → Page_Load → Page_PreRender の3行のみ(クリックイベントはまだない)
「PostBack を発生させる」押下後 追加された行 Page_Init → Page_Load → ReloadButton_Click → Page_PreRender の順で追加
ボタンを複数回押したとき ログが消えずに積み上がるか ViewState 保存により、Page が作り直されても履歴が残る

補足:フッターの「現在時刻」「IsPostBack」も同じ仕組みで毎回更新されている(Site.Master.cs)

よくある誤解・注意点

  • 「フィールドに値を保持しておける」は誤解 → 実際は毎回破棄される。保持するには ViewState / Session
  • PreRender より後でコントロールの値を変えても、表示に反映されないことがある
  • イベントハンドラー(Click)は Load の後・PreRender の前 に実行される順序を忘れずに

まとめ

  • Page はリクエストごとに使い捨てのオブジェクト
  • 実行順序:Init → Load → イベント → PreRender
  • 状態を持ち越すには ViewState / Session など明示的な保存が必要 → 次章以降で深掘り
