状態管理

Session / Application / Cache の使い分け

本日のゴール

  • ViewState / Session / Application / Cache の「共有される範囲」の違いを説明できる
  • Session がユーザーごとに独立し、Application が全ユーザー共通であることを、実際に複数ブラウザで確認する
  • Cache の有効期限の考え方を説明できる

前章の疑問

第4章:ViewState はページ単位・クライアント保存だった。では「ログインユーザー情報」や「全ユーザー共通のお知らせ」はどこに置くのか?

定義:状態保存の3つの方式

Session:
ユーザーごとの状態をサーバーメモリに保存。同じユーザーが複数のページにアクセスしても情報が共有される。ログイン状態・カート内容など個人情報向け。
Application:
全ユーザー共通の状態をサーバーメモリに保存。アプリケーション全体で共有される。アクセスカウンター・共通設定など。複数リクエストの並列アクセスに注意(Lock/Unlock 必須)。
Cache:
期限付きの一時保存。再利用したいデータを保存し、有効期限や依存関係でキャッシュを管理。同じデータベースクエリを何度も実行しないために使う。

4つの状態保持の範囲

ViewState

単位:ページ

同一ページ内の状態

Session

単位:ユーザー

ログインユーザーごと

Application

単位:アプリ全体

全ユーザー共通

Cache

単位:アプリ全体

期限付き一時保存

右に行くほど「共有範囲が広く」「影響力も大きい」— 何を書き込むかは慎重に。

コードで確認

05_StateManagement.aspx.cs

Session["Counter"] = ((int?)Session["Counter"] ?? 0) + 1;

Cache.Insert("CachedMessage",
    $"Cache に保存しました: {DateTime.Now:HH:mm:ss}",
    null, DateTime.Now.AddMinutes(5), TimeSpan.Zero);

Global.asax.cs

protected void Application_BeginRequest(object sender, EventArgs e)
{
    Application.Lock();
    Application["TotalRequests"] = (int)Application["TotalRequests"] + 1;
    Application.UnLock();
}

デモで確認するポイント

いつどこを見る何が分かるか
「Session カウント +1」押下Session[Counter]自分のブラウザだけで増える
別ブラウザ(プライベートウィンドウ)で同じページそこの Session[Counter]0から独立してカウントされる = ユーザーごとに別管理
誰かがページを開くたびApplication[TotalRequests]全ユーザー共通で増加(Global.asax.cs)
5分待ってから再読み込みCache[CachedMessage]有効期限切れで「未設定」に戻る

使い分けの目安

種類単位用途
ViewStateページ同一ページ内のコントロール状態
Sessionユーザーログインユーザーごとの情報(ログイン状態、カート内容など)
Applicationアプリ全体全ユーザー共通の値・カウンタ
Cacheアプリ全体再利用したいデータの一時保存(有効期限あり)

よくある誤解・注意点

  • Session は「サーバーが覚えている」仕組み — InProc モードでは アプリ再起動で消える(Web.config: sessionState timeout="20")
  • Application への書き込みは複数リクエストが同時に来るため Lock() / UnLock() が必要
  • Cache は「必ず残っている」保証がない(期限切れ・メモリ不足で消えることがある)前提で設計する

まとめ

  • ViewState(ページ)→ Session(ユーザー)→ Application/Cache(アプリ全体)の順に共有範囲が広がる
  • Global.asax.cs の Application_Start / Application_BeginRequest はすべてのページ・全リクエストの裏で動いている
  • 用途に応じて適切な保存場所を選ぶことが重要
