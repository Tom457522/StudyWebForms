サーバーコントロール

runat="server" が生む「サーバー側から触れる HTML」

本日のゴール

  • runat="server" を付けたコントロールが、コードビハインドから同名プロパティとして扱えることを説明できる
  • サーバーコントロールが最終的に普通の HTML タグに変換されることを、実際のレンダリング結果で確認する
  • Server.HtmlEncode による基本的な XSS 対策の考え方を説明できる

前章の振り返り

第1章:Page は毎リクエストで作り直され、決まった順序でイベントが実行される

今回は「そのPageの中で、コードから触っている部品(コントロール)の正体」を見ていきます。

コントロールの正体

<asp:Label runat="server">
<span>...</span>
<asp:TextBox runat="server">
<input type="text">
  • runat="server" が付いた要素だけが、コードビハインドから 同じIDのプロパティとして参照できる
  • Web Forms の ID は、最終的な HTML の id 属性にそのまま(または命名規則に沿って)反映される

コードで確認①:マークアップ

02_ServerControls.aspx

<asp:Label ID="NameLabel" runat="server"
    AssociatedControlID="NameTextBox" Text="名前" />
<asp:TextBox ID="NameTextBox" runat="server" />
<asp:Button ID="ReflectButton" runat="server"
    Text="ラベルに反映" OnClick="ReflectButton_Click" />
<asp:Label ID="ResultLabel" runat="server"
    Text="ここに結果が表示されます" />

コードで確認②:コードビハインド

02_ServerControls.aspx.cs

protected void ReflectButton_Click(object sender, EventArgs e)
{
    ResultLabel.Text = string.IsNullOrWhiteSpace(NameTextBox.Text)
        ? "未入力です。TextBox.Text は空です。"
        : $"こんにちは、{Server.HtmlEncode(NameTextBox.Text)} さん。" +
          "TextBox.Text を Label.Text に反映しました。";
}

NameTextBox / ResultLabel はマークアップの ID がそのままプロパティ名になっている点に注目

セキュリティの基本:HtmlEncode

もし HtmlEncode しなかったら

入力値をそのまま HTML に埋め込むと、<script> タグなどが実行されてしまう(XSS)おそれがあります。

  • Server.HtmlEncode は < > & などを無害な文字列に変換する
  • ユーザー入力を画面に表示する処理では、基本的に必ず通す

デモで確認するポイント

いつどこを見る何が分かるか
「ラベルに反映」押下後結果ラベルの文言ReflectButton_Click が TextBox の値を読み取り Label に書き込んでいる
ブラウザの「ページのソースを表示」出力された HTML タグasp:Label → <span>、asp:TextBox → <input> に変換されている
HTML タグの id 属性id="NameTextBox" などWeb Forms の ID が HTML の id にそのまま反映されている

よくある誤解・注意点

  • 「サーバーコントロール」という特別な実行モデルがあるわけではない → 最終的にはただの HTML
  • runat="server" を忘れると、コードビハインドから参照できない普通の HTML タグになる
  • ユーザー入力を表示するときは HtmlEncode を忘れずに

まとめ

  • runat="server" の要素はコードビハインドから同じ ID のプロパティとして操作できる
  • レンダリング結果は最終的に普通の HTML(<span> / <input> など)
  • ユーザー入力の表示には Server.HtmlEncode を使う
